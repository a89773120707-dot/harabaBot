# AI_WORK_RULES.md — Правила работы ИИ над Haraba Mini

## Главная цель

Работать над Haraba Mini строго по утверждённому плану, без хаотичных фиксов, без догадок и без самостоятельных deploy/restart/push.

ИИ должен не "угадывать", а доказывать проблему фактами.

---

# 1. Главный принцип работы

Любая задача выполняется только по схеме:

```text
1. Диагностика
2. Факты
3. План
4. Утверждение пользователя
5. Реализация
6. Проверка
7. Отчёт
```

Запрещено переходить к следующему шагу, пока предыдущий не подтверждён.

---

# 2. Что ИИ НЕ должен делать

Запрещено без отдельного утверждения пользователя:

```text
git commit
git push
git reset --hard
deploy на VPS
systemctl restart
pkill / kill -9
nohup / setsid
SCP файлов на VPS
ручное редактирование файлов на VPS
очистка БД
очистка dedup
изменение cron
изменение .env
изменение токенов
```

Если нужно сделать одно из этих действий — сначала остановиться и запросить подтверждение.

---

# 3. Запрет на догадки

Запрещённые формулировки:

```text
скорее всего
возможно
наверное
похоже
думаю что причина
```

Вместо этого писать:

```text
Доказано:
...

Не доказано:
...

Нужна проверка:
...
```

Любой вывод должен быть подтверждён:

```text
логом
SQL-запросом
git diff
git status
systemctl status
journalctl
результатом теста
```

---

# 4. Правило одного изменения

За один блок можно чинить только одну проблему.

Пример:

```text
если чиним needs_comment
не трогаем config_name
не трогаем get_enabled_recipients
не трогаем cron
не трогаем dedup
```

Если обнаружена новая проблема — записать её в "следующие задачи", но не чинить сразу.

---

# 5. Правило Git

GitHub — единственный источник правды.

Запрещено:

```text
копировать файлы на VPS через SCP
редактировать код на VPS вручную
делать hotfix напрямую на сервере
```

Правильный порядок:

```text
локальный diff
локальные тесты
commit
push
VPS git fetch
VPS git reset --hard origin/main
проверка
restart только после утверждения
```

---

# 6. Правило перед commit

Перед любым commit обязательно показать:

```bash
git status --short
git diff --name-only
git diff <изменяемый_файл>
```

ИИ обязан явно написать:

```text
В commit попадут файлы:
...

Не попадут:
...
```

Если в `git status` есть лишние файлы:

```text
.env
*.db
logs/*
results/*
vps_*.py
block_*.py
check_*.py
debug scripts
```

их нельзя добавлять в commit.

---

# 7. Правило перед push

Перед push обязательно показать:

```bash
git show --stat HEAD
git diff origin/main..HEAD --name-only
git log --oneline -3
```

Push разрешён только если пользователь утвердил.

---

# 8. Правило перед deploy

Перед deploy на VPS обязательно:

```text
1. Показать commit hash локально
2. Показать origin/main
3. Показать что именно будет обновлено на VPS
4. Получить отдельное подтверждение пользователя
```

На VPS выполнять только:

```bash
cd /home/haraba/harabaBot_code
git fetch origin
git reset --hard origin/main
```

После этого показать:

```bash
git log --oneline -3
git status --short
```

---

# 9. Правило restart

Restart делать только после deploy и отдельного подтверждения.

Если нужно перезапустить feedback bot:

```bash
systemctl restart haraba-feedback-bot.service
```

Если нужно перезапустить admin bot:

```bash
systemctl restart haraba-admin-bot.service
```

Запрещено:

```bash
pkill -9
nohup
setsid
ручной запуск второго процесса
```

После restart показать:

```bash
systemctl status <service> --no-pager
journalctl -u <service> -n 80 --no-pager
```

---

# 10. Правило диагностики ошибки

Если что-то не работает, порядок такой:

## 1. Симптом

Написать:

```text
Что именно не работает?
Когда началось?
Что ожидали?
Что получили?
```

## 2. Логи

Показать релевантные логи.

## 3. БД

Показать SQL-запросы, если проблема связана с данными.

## 4. Код

Показать точные функции и импорты.

## 5. Вывод

Только после этого писать причину.

---

# 11. Правило Telegram bot

Если проблема с кнопками/реакциями:

Проверять строго по цепочке:

```text
Telegram callback дошёл?
↓
handler вызвался?
↓
ошибка в логах есть?
↓
feedback записался?
↓
reaction_details записался?
↓
reason_code корректный?
```

Не искать typo, пока не доказано, что callback дошёл до handler.

---

# 12. Правило pipeline

Если карточки не приходят:

Проверять строго:

```text
cron работает?
pipeline запускался?
flock не висит?
send_ready сколько?
sent сколько?
skipped сколько?
failed сколько?
sent_ads пополнился?
telegram_sender_report что показывает?
```

Не чистить dedup без backup и подтверждения.

---

# 13. Правило БД

Перед любым изменением БД:

```text
1. Сделать backup
2. Показать COUNT до
3. Выполнить изменение
4. Показать COUNT после
5. Проверить данные
```

Запрещено чистить:

```text
feedback
reaction_details
telegram_users
```

без отдельного явного разрешения.

---

# 14. Приоритеты проекта

Текущий порядок работ:

```text
P0:
1. Проверить get_enabled_recipients
2. Закрыть Block 2 config_name
3. Убедиться что реакции стабильно сохраняются

P1:
4. config_report
5. learning_dashboard
6. config_diagnostics

P2:
7. near_miss analysis
8. config_suggestions
9. config versioning

P3:
10. manager search requests
11. voice/photo requests
12. Auto.ru / Avito
```

Запрещено начинать P2/P3, пока P0 не закрыт.

---

# 15. Текущая флагманская идея проекта

Haraba Mini — это не просто рассылка карточек.

Цель:

```text
рынок
→ карточки
→ реакции менеджеров
→ аналитика конфигов
→ предложения улучшений
→ owner подтверждает
→ система становится точнее
```

Бот ничего не меняет автоматически.

Owner всегда принимает решение.

---

# 16. Формат отчёта после каждого блока

После каждого блока ИИ обязан дать отчёт:

```text
Блок:
Цель:
Что сделано:
Что проверено:
Результат:
Что НЕ трогали:
Следующий шаг:
Нужно ли утверждение пользователя:
```

---

# 17. Правило остановки

Если во время работы появилась новая ошибка:

```text
СТОП.
Новая проблема обнаружена.
Текущий блок не продолжаю.
Сначала показываю факты и жду решения пользователя.
```

---

# 18. Главная ошибка, которую нельзя повторять

Нельзя бегать за неподтверждёнными гипотезами.

Пример плохого поведения:

```text
думаю typo
исправляю typo
деплою
потом оказывается проблема была ImportError
```

Правильное поведение:

```text
сначала доказать путь:
callback → handler → import → DB
только потом чинить конкретную ошибку
```

---

# 19. Финальное правило

Если пользователь не сказал "утверждаю" — не выполнять опасное действие.

Опасные действия:

```text
commit
push
deploy
restart
kill
db delete
cron change
.env change
```

Сначала план.

Потом утверждение.

Потом действие.
